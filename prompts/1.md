### 4. Handling Offline Sync (Node.js + WatermelonDB)

You need two endpoints on your Express backend that follow the **Push/Pull** pattern.

**Endpoint A: `GET /sync/pull`**
*   **Input:** `last_pulled_at` (timestamp).
*   **Logic:**
    1.  Find all records (products, variants, orders) where `updated_at > last_pulled_at` AND `tenant_id = user.tenant_id`.
    2.  Return generic JSON: `{ changes: { products: { created: [], updated: [], deleted: [] } ... }, timestamp: NOW() }`

**Endpoint B: `POST /sync/push`**
*   **Input:** JSON of changes made locally on the device.
*   **Logic:**
    1.  Loop through the changes.
    2.  Execute SQL Transaction.
    3.  **Conflict Resolution:** If the server record is newer than the mobile record, usually "Server Wins" or you apply the mobile change only if fields don't conflict.

**Node.js Pseudocode for Pull:**
```javascript
app.get('/sync', authMiddleware, async (req, res) => {
    const { lastPulledAt } = req.query;
    const tenantId = req.user.tenantId;

    // Fetch changes since last sync
    const createdProducts = await db.query(
        "SELECT * FROM products WHERE tenant_id = $1 AND created_at > $2", 
        [tenantId, lastPulledAt]
    );
    
    const updatedProducts = await db.query(
        "SELECT * FROM products WHERE tenant_id = $1 AND updated_at > $2 AND created_at <= $2", 
        [tenantId, lastPulledAt]
    );

    res.json({
        changes: {
            products: {
                created: createdProducts.rows,
                updated: updatedProducts.rows,
                deleted: [], // You need a Soft Delete mechanism to track deletions
            }
        },
        timestamp: new Date().toISOString()
    });
});
```

This is a smart architectural decision. By decoupling your modules, you avoid "spaghetti code" where a change in the Sales logic accidentally breaks your Inventory calculations.

To achieve this in a **Node.js/Express Monolith**, you should use an **Event-Driven Architecture (EDA)**.

Here is how you implement strict isolation using a **Pub/Sub (Publish/Subscribe)** pattern.

### The Core Concept
1.  **Sales Module:** "I just completed Order #101." (Emits Event) -> *Job Done.*
2.  **Purchase Module:** "I just received PO #500." (Emits Event) -> *Job Done.*
3.  **Inventory Module:** Listens for specific events. When it hears "Order Completed," it runs its own logic to deduct stock.

```javascript
async function createOrder(tenantId, userId, orderData) {
    // 1. Start Transaction
    const client = await db.connect();
    try {
        await client.query('BEGIN');

        // 2. Create Order Header
        const orderResult = await client.query(
            `INSERT INTO orders (tenant_id, user_id, total_amount) VALUES ($1, $2, $3) RETURNING id`,
            [tenantId, userId, orderData.total]
        );
        const orderId = orderResult.rows[0].id;

        // 3. Create Order Items
        for (const item of orderData.items) {
            await client.query(
                `INSERT INTO order_items (order_id, variant_id, quantity) VALUES ($1, $2, $3)`,
                [orderId, item.variantId, item.quantity]
            );
        }

        await client.query('COMMIT');

        // 4. FIRE AND FORGET (Decoupled)
        // We pass the entire payload so Inventory doesn't have to query the DB again immediately
        eventBus.emit(EVENTS.ORDER_CREATED, {
            tenantId,
            orderId,
            items: orderData.items // [{ variantId: '...', quantity: 5 }]
        });

        return { success: true, orderId };

    } catch (e) {
        await client.query('ROLLBACK');
        throw e;
    } finally {
        client.release();
    }
}
```

### Critical Consideration: Data Integrity
Since the modules are isolated, what happens if the **Sale succeeds** but the **Inventory update fails** (e.g., database lock)?
1.  **Transactional Outbox (Advanced):** Save the event to a DB table (`outbox_events`) *inside* the Sales transaction. A separate worker reads that table and processes the inventory update.
2.  **Retries (Easier):** If using a queue like BullMQ, it automatically retries failed jobs.
3.  **Reconciliation:** Since you have an `inventory_log`, you can periodically run a script to ensure `sum(log) == stock_quantity`.

can you expand on the transactional outbox implementation?

The **Transactional Outbox** pattern solves the "Dual Write Problem."

Without it, if your database transaction commits (Order Saved) but your app crashes before the event is emitted, your Inventory module never knows about the sale. The data becomes inconsistent.

With the Outbox pattern, you save the **Event** into the database **inside the same transaction** as the **Order**. This guarantees atomicity: **Either both happen, or neither happens.**

Here is the implementation in **Node.js + PostgreSQL**.

---

### 1. The Database Schema
Create a table to act as your "Outbox."

```sql
CREATE TABLE outbox (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    aggregate_type VARCHAR(50) NOT NULL, -- e.g., 'ORDER', 'PURCHASE'
    aggregate_id UUID NOT NULL,          -- e.g., The Order ID
    event_type VARCHAR(50) NOT NULL,     -- e.g., 'ORDER_CREATED'
    payload JSONB NOT NULL,              -- The actual data needed by subscribers
    
    created_at TIMESTAMPTZ DEFAULT NOW(),
    processed_at TIMESTAMPTZ,            -- Null = Not sent yet
    
    -- Optional: Retry logic
    retry_count INT DEFAULT 0,
    last_error TEXT
);

-- Index for fast polling
CREATE INDEX idx_outbox_unprocessed ON outbox(created_at) WHERE processed_at IS NULL;
```

---

### 2. The Producer (Sales Module)
In your Sales service, instead of emitting the event directly, you insert it into the `outbox` table using the **same client/transaction**.

**`src/modules/sales/sales.service.js`**
```javascript
const db = require('../../shared/database');
const { EVENTS } = require('../../shared/event-bus');

async function createOrder(tenantId, userId, orderData) {
    const client = await db.connect();
    
    try {
        await client.query('BEGIN'); // Start Transaction

        // 1. Insert Order
        const orderResult = await client.query(
            `INSERT INTO orders (tenant_id, user_id, total_amount) VALUES ($1, $2, $3) RETURNING id`,
            [tenantId, userId, orderData.total]
        );
        const orderId = orderResult.rows[0].id;

        // 2. Insert Order Items... (omitted for brevity)

        // 3. INSERT INTO OUTBOX (The "Event" is saved as data)
        const eventPayload = {
            tenantId,
            orderId,
            items: orderData.items,
            timestamp: new Date().toISOString()
        };

        await client.query(
            `INSERT INTO outbox (aggregate_type, aggregate_id, event_type, payload)
             VALUES ($1, $2, $3, $4)`,
            ['ORDER', orderId, EVENTS.ORDER_CREATED, JSON.stringify(eventPayload)]
        );

        // 4. Commit (Order and Event are now both safely in DB)
        await client.query('COMMIT');
        
        return { success: true, orderId };

    } catch (e) {
        await client.query('ROLLBACK'); // If this happens, neither the Order nor the Event exists
        throw e;
    } finally {
        client.release();
    }
}
```

### 3. The Relay (The Message Broker)
Now you need a background process that checks the `outbox` table and pushes messages to your Event Bus (or Redis/RabbitMQ).

**Why `FOR UPDATE SKIP LOCKED`?**
This is a specific PostgreSQL feature. It allows you to run multiple instances of your Node app (scaling) without them trying to process the same event twice.

**`src/shared/outbox-relay.js`**
```javascript
const db = require('./database');
const { eventBus } = require('./event-bus');

async function processOutbox() {
    const client = await db.connect();
    
    try {
        await client.query('BEGIN');

        // 1. Fetch unprocessed events (Lock them so other workers don't grab them)
        const result = await client.query(
            `SELECT * FROM outbox 
             WHERE processed_at IS NULL 
             ORDER BY created_at ASC 
             LIMIT 50 
             FOR UPDATE SKIP LOCKED` 
        );

        if (result.rows.length === 0) {
            await client.query('COMMIT');
            return; // Nothing to do
        }

        // 2. Publish to the Event Bus
        for (const row of result.rows) {
            try {
                // Determine topic and emit
                console.log(`[Relay] Publishing ${row.event_type} for ID: ${row.aggregate_id}`);
                
                // This emits to your internal listeners (Inventory Module)
                eventBus.emit(row.event_type, row.payload);

                // 3. Mark as processed
                await client.query(
                    `UPDATE outbox SET processed_at = NOW() WHERE id = $1`,
                    [row.id]
                );
            } catch (err) {
                console.error(`Failed to publish event ${row.id}`, err);
                // Optional: Increment retry_count here
            }
        }

        await client.query('COMMIT');

    } catch (e) {
        await client.query('ROLLBACK');
        console.error("Outbox Relay Error:", e);
    } finally {
        client.release();
    }
}

// Run the relay every 2 seconds
function startOutboxRelay() {
    setInterval(processOutbox, 2000);
}

module.exports = { startOutboxRelay };
```

### 4. The Consumer (Inventory Module) & Idempotency
The Outbox pattern guarantees **"At Least Once"** delivery.
*   *Scenario:* The Relay emits the event, but the database connection dies before it can mark `processed_at = NOW()`.
*   *Result:* The Relay will pick up the same event again 2 seconds later.

Therefore, your Consumers (Inventory Module) must be **Idempotent**. They must check if they have already processed this specific event.

**`src/modules/inventory/inventory.service.js`**
```javascript
const db = require('../../shared/database');

async function deductStock(tenantId, items, orderId) {
    const client = await db.connect();
    try {
        await client.query('BEGIN');

        // 1. IDEMPOTENCY CHECK
        // Check if we already have an inventory log for this specific Order ID
        const check = await client.query(
            `SELECT 1 FROM inventory_log WHERE reference_id = $1 AND reason = 'sale'`,
            [orderId]
        );

        if (check.rowCount > 0) {
            console.log(`[Inventory] Order ${orderId} already processed. Skipping.`);
            await client.query('ROLLBACK');
            return; 
        }

        // 2. If not processed, proceed with logic...
        for (const item of items) {
            await client.query(
                `INSERT INTO inventory_log ...`, [...]
            );
            await client.query(
                `UPDATE variants ...`, [...]
            );
        }

        await client.query('COMMIT');
    } catch (e) {
        await client.query('ROLLBACK');
        throw e;
    } finally {
        client.release();
    }
}
```

### Summary of Benefits
1.  **Reliability:** You never lose a sale event.
2.  **Order:** `ORDER BY created_at ASC` ensures Inventory processes orders in the exact sequence they happened.
3.  **Decoupling:** If the Inventory module throws an error, the `outbox` record stays unprocessed (or you can add a `failed` status), allowing you to retry later without disturbing the Sales module.

what all does the journal ledger track

The **Inventory Ledger (or Stock Card)** is the single most important table in your entire database. It is the "Source of Truth."

If your `products` table says you have 10 items, but the Ledger says you should have 12, **the Ledger is right** and your product table is wrong.

Here is exactly what the ledger must track to handle **Retail, Grocery, and Multi-location** scenarios efficiently.

---

### 1. The Core Data Points
Every row in the `inventory_ledger` table represents a physical movement of goods.

| Field | Description | Why it is needed |
| :--- | :--- | :--- |
| **Direction** | `IN` vs `OUT` | Simple math: `IN` adds to stock, `OUT` subtracts. |
| **Quantity** | e.g., `+5` or `-1.5` | Supports integers (Retail) and decimals (Grocery - kg/liters). |
| **Running Balance** | e.g., `45` | The stock level *after* this transaction. (Crucial for historical reporting). |
| **Cost Basis** | Cost per unit *at that moment*. | Required for **FIFO/Weighted Average** profit calculations. |
| **Reference** | `order_id`, `po_id`, `adjustment_id` | Links the movement to the document that caused it. |

---

### 2. Comprehensive Tracking List
You need to support these specific "Reason Codes" to handle real-world small business chaos.

#### A. Transactional Movements
1.  **Sales (OUT):** Decrements stock when a customer buys.
2.  **Sales Return (IN):** Increments stock when a customer brings an item back (restockable).
3.  **Purchase Receive (IN):** Increments stock when a vendor delivers goods.
4.  **Purchase Return (OUT):** Decrements stock when you send damaged goods back to the vendor.

#### B. Internal Movements
5.  **Transfer Out (OUT):** Moving stock *from* Warehouse A.
6.  **Transfer In (IN):** Receiving stock *at* Warehouse B.
    *   *Note:* These usually come in pairs.
7.  **Manufacturing/Bundling (Bundle Logic):**

### 4. Why "Cost Price" is Critical here
This is where most junior devs fail. You **must** track the cost price in the ledger, not just on the product table.

**Scenario:**
1.  **Jan 1:** You buy 10 Apples at **$1.00**. (Product Cost = $1.00)
2.  **Jan 2:** You buy 10 Apples at **$1.50** (Inflation). (Product Cost = $1.50)
3.  **Jan 3:** You sell 5 Apples.

*   *Question:* What was the cost of those 5 apples? $1.00 or $1.50?
*   *Answer:* Using **FIFO (First-In-First-Out)**, the cost was $1.00. Using **AVCO (Average Cost)**, it was $1.25.

If you don't store the cost in the ledger history, you cannot calculate accurate Profit & Loss at the end of the month. You will only know the *current* cost, which distorts historical profit.

### 3. The "Golden Triangle" of a Transaction
This is where your event-driven architecture shines. A single **Sale Event** might trigger writes to **all three ledgers** depending on the context.

#### Scenario A: Cash Sale ($100)
1.  **Inventory Ledger:** Stock `-1` (Item leaves).
2.  **Financial Ledger:** Cash `+100` (Money enters Cash Drawer).
3.  **Party Ledger:** No entry (Settled immediately).

#### Scenario B: Credit Sale ($100)
1.  **Inventory Ledger:** Stock `-1` (Item leaves).
2.  **Financial Ledger:** **No Entry** (No actual money moved yet!).
3.  **Party Ledger:** Customer Balance `+100` (Customer owes money).

#### Scenario C: Customer Pays their Debt later
1.  **Inventory Ledger:** No entry.
2.  **Financial Ledger:** Cash `+100`.
3.  **Party Ledger:** Customer Balance `-100` (Debt is settled).

### 4. How to Calculate Profit (The Logic)
Because you have separated these ledgers, calculating "Real Profit" is a specific query, not just looking at the bank balance.

**Gross Profit = (Sales Revenue) - (Cost of Goods Sold)**
*   *Sales Revenue:* Sum of `finance_transactions` where category = 'Sales'.
*   *COGS:* Sum of `inventory_ledger` (cost_price * quantity) for all sales.

**Net Profit = Gross Profit - Expenses**
*   *Expenses:* Sum of `finance_transactions` where category_type = 'expense' (Rent, Electricity, etc.).

### Summary of Ledgers needed for your SaaS
1.  **Inventory Ledger:** Tracks **Quantities** & **COGS** (The "Stuff").
2.  **Financial Ledger:** Tracks **Cash Flow** & **Expenses** (The "Real Money").
3.  **Party Ledger:** Tracks **Debt** & **Credit** (The "Promises").

If you implement these three, your SaaS will handle everything from a simple T-shirt stall (Cash only) to a B2B wholesaler (Credit/Invoicing) without changing the database schema.

```javascript
// This function creates a bare-minimum product
async function getOrCreateProduct(tenantId, itemData, client) {
    // 1. If variant_id exists, just verify it belongs to tenant
    if (itemData.variant_id) {
        return itemData.variant_id;

    const productRes = await client.query(

        [tenantId, itemData.name]

    // B. Create the Default Variant
    const variantRes = await client.query(
        `INSERT INTO variants 
         (tenant_id, product_id, name, sku, cost_price, selling_price, track_inventory) 
         VALUES ($1, $2, 'Standard', $3, $4, $5, true) 
         RETURNING id`,

            itemData.cost_price, 
            itemData.selling_price || (itemData.cost_price * 1.5) // Default 50% margin if missing

}
```

**`src/modules/purchases/purchase.service.js` (The Orchestrator)**
Now, use that function inside your purchase flow.

```javascript
const db = require('../../shared/database');
const { eventBus, EVENTS } = require('../../shared/event-bus');
const productService = require('../products/product.service'); // Importing the service, not the DB model

async function createPurchaseOrder(tenantId, data) {
    const client = await db.connect();
    
    try {
        await client.query('BEGIN');

        const resolvedItems = [];

        // STEP 1: RESOLVE ITEMS (The Magic)
        for (const item of data.items) {
            // This line abstracts the complexity of "Does it exist?"
            const variantId = await productService.getOrCreateProduct(tenantId, item, client);
            
            resolvedItems.push({
                variantId: variantId,
                quantity: item.quantity,
                cost: item.cost_price
            });
        }

        // STEP 2: CREATE PO HEADER
        const poRes = await client.query(
            `INSERT INTO purchase_orders (tenant_id, vendor_id, status) VALUES ($1, $2, 'received') RETURNING id`,
            [tenantId, data.vendor_id]
        );
        const poId = poRes.rows[0].id;

        // STEP 3: CREATE PO ITEMS
        for (const item of resolvedItems) {
            await client.query(
                `INSERT INTO purchase_items (purchase_id, variant_id, quantity, unit_cost) VALUES ($1, $2, $3, $4)`,
                [poId, item.variantId, item.quantity, item.cost]
            );
        }

        // STEP 4: EMIT EVENT (For Inventory Module to pick up)
        // Notice we emit 'PURCHASE_RECEIVED'. 
        // The Inventory module creates the stock ledger entries.
        // It doesn't care if the product is 1 second old or 1 year old.
        eventBus.emit(EVENTS.PURCHASE_RECEIVED, {
            tenantId,
            purchaseId: poId,
            items: resolvedItems
        });

        await client.query('COMMIT');
        return { success: true, purchaseId: poId };

    } catch (e) {
        await client.query('ROLLBACK');
        throw e;
    } finally {
        client.release();
    }
}
```

3.  **Logical Schemas:** Use PostgreSQL **Schemas** to physically separate the tables.

### 3. How to Query Without Joins?

You might be thinking: *"If I don't have Foreign Keys, how do I display the order history with product details?"*

**Approach A: The Snapshot (Recommended)**
Since `sales.order_items` now stores `product_name_snapshot`, you don't *need* to join with the Product table to show a receipt. You have all the data you need right there in the Sales module. This makes reads incredibly fast.

**Approach B: Data Aggregation (API Composition)**
If you are displaying a dashboard and need data from two modules (e.g., "Top Selling Items" + "Current Stock Level"), you do it in code, not SQL.

*   **Bad (Monolith):** `SELECT * FROM sales JOIN inventory ...`
*   **Good (Isolated):**

```javascript
// Analytics Controller (Or a specific BFF - Backend for Frontend)
async function getTopSellingWithStock(tenantId) {
    
    // 1. Ask Sales Module: What are the top selling product IDs?
    // Returns: [{ productId: 'uuid-1', sold: 50 }, { productId: 'uuid-2', sold: 40 }]
    const salesStats = await salesService.getTopSellers(tenantId); 
    
    // 2. Extract IDs
    const productIds = salesStats.map(s => s.productId);
    
    // 3. Ask Inventory Module: What is the stock for these IDs?
    // Returns: [{ productId: 'uuid-1', stock: 5 }, { productId: 'uuid-2', stock: 100 }]
    const stockLevels = await inventoryService.getStockForIds(tenantId, productIds);

    // 4. Merge in Code
    return salesStats.map(sale => ({
        ...sale,
        stock: stockLevels.find(i => i.productId === sale.productId)?.stock || 0
    }));
}
```

### 4. Handling "Referential Integrity" without Foreign Keys

Since you removed the database constraints, the application layer must now take responsibility.

**Problem:** What if I delete a Product in the Product Module, but the Inventory Module still thinks I have 50 in stock?

**Solution: Asynchronous Cleanup (Event Driven)**

1.  **Product Module:** User deletes a product.
    *   Database: `DELETE FROM products.catalog WHERE id = 'xyz'`
    *   Event Bus: Emits `PRODUCT_DELETED` event (`{ id: 'xyz' }`).
2.  **Inventory Module:** Listens to `PRODUCT_DELETED`.
    *   Action: `DELETE FROM inventory.stock_levels WHERE product_id = 'xyz'` (or archive it).
3.  **Sales Module:** Listens to `PRODUCT_DELETED`.
    *   Action: **Do nothing.** (Because sales history is immutable. You sold it, you keep the record).

### Summary of Database Changes for Isolation
1.  **Schemas:** Use `products`, `sales`, `inventory` schemas.
2.  **Tables:** Remove `REFERENCES` that cross schema boundaries.
3.  **Columns:** Add `_snapshot` columns (name, price, sku) to `sales.order_items` and `purchase.items`.
4.  **Application:** Perform "UI Composition" or "Service Aggregation" instead of SQL JOINS.

should payments be its own module or split into the sales and purchases modules to handle payment in and out?

Given your requirement for **strict module isolation** and the nature of small business operations (Point of Sale, Expenses, and Credit/Khata), **Payments must be a standalone module.**

Do **not** split it into Sales and Purchases.

### Why a Dedicated Payments Module?

1.  **The "Split Payment" Problem:**
    *   A customer buys $100 worth of goods. They pay $50 in Cash and $50 via Credit Card.
    *   If payment logic is inside the Sales module, you have to build complex logic there to handle multiple payment rows per order.
2.  **The "Debt" Problem (Khata):**
    *   A customer buys items on Monday (Sales Module active).
    *   They come back on Friday to pay the debt.
    *   This transaction is **not** a new Sale. It is just a Payment. If payments are locked inside `sales`, you have to create a "Fake Sale" to record the money. That ruins your inventory and revenue stats.
3.  **The "Expense" Problem:**
    *   You pay Rent. This is "Money Out." If you only have payments in `purchases`, where do you record Rent? You would need a third payment table in `expenses`.
4.  **Single Integration Point:**
    *   You only want to write the code for Stripe, PayPal, Razorpay, or Square **once**.

### 2. The Workflow (Inter-Module Communication)

Here is how the Sales module talks to the Payment module without strict coupling.

#### Scenario: A Cash Sale

1.  **Frontend:** Sends payload to `POST /sales/orders`.
    ```json
    {
      "items": [...],
      "payment": { "method": "cash", "amount": 100 }
    }
    ```
2.  **Sales Service:**
    *   Calculates total: $100.
    *   Creates Order in `sales.orders` (Status: `pending_payment`).
    *   **Synchronous Call (Internal API):** Calls `paymentService.processPayment(...)`.
3.  **Payment Service:**
    *   Records +$100 in `payments.transactions`.
    *   Returns `success: true` and a `transactionId`.
4.  **Sales Service:**
    *   Updates Order Status to `paid`.
    *   Commits Transaction.

**Code Example (Sales Controller):**

```javascript
    // 1. Create Order (Pending)
    const orderId = await createPendingOrder(data); 
    
    try {
        // 2. Process Payment (The Handshake)
        const paymentResult = await paymentService.recordTransaction({
            tenantId: data.tenantId,
            amount: data.totalAmount, // +100
            methodId: data.paymentMethodId,
            referenceId: orderId,
            referenceType: 'sales.order'
        });

        // 3. Confirm Order
        await markOrderAsPaid(orderId);
        
        return { orderId, paymentId: paymentResult.id };
        
    } catch (err) {
        // Payment failed? Cancel order.
        await cancelOrder(orderId); 
        throw new Error("Payment Failed");
    }
}
```

This is where the isolated module shines.

*   There is **NO** Sale here. No inventory moves.

2.  **Purchases Module:** Calls `paymentService.recordTransaction`.
3.  **Payments Module:**

By isolating Payments, you solve the "Many-to-Many" problem:
*   **One Sale** can have **Multiple Payments** (Split tender).
*   **One Payment** can cover **Multiple Sales** (Paying off a monthly tab).

If you had split payments into Sales/Purchases, enabling a customer to "Pay off their monthly tab" would have required complex queries across multiple tables. With a dedicated module, it's a simple `INSERT`.

In a **Strictly Isolated Modular Monolith**, having `sales.service.js` directly import `payment.service.js` **is a violation of decoupling**.

If you later decide to rip the Payments module out into a separate microservice, that `require('../payments/...')` line will break your application.

However, we have a dilemma: **User Experience (UX) vs. Architecture.**
*   **Inventory:** Can be "Eventually Consistent." If the stock updates 200ms later, no one cares. -> **Events are perfect.**
*   **Payments:** Must be "Immediately Consistent." You cannot tell the customer "Purchase Successful" if the credit card hasn't actually been charged yet. -> **Events are too slow/complex** (requires WebSockets/Polling).

Here is how we solve this while maintaining strict isolation: **The Orchestrator Pattern (or Use Cases).**

---

### The Solution: "The Checkout Orchestrator"

Instead of the **Sales Module** calling the **Payment Module**, you create a **higher-level workflow** that sits *above* both of them.

This "Orchestrator" is the puppet master. It imports both modules, coordinates the transaction, and handles the failure. Neither module knows the other exists.

#### Directory Structure Update
Add a `workflows` or `use-cases` folder.

```text
/src
  /modules
    /sales      <-- Knows ONLY about Orders
    /payments   <-- Knows ONLY about Money
    /inventory  <-- Knows ONLY about Stock
  /workflows    <-- Knows EVERYTHING (The "Glue")
    checkout.workflow.js
```

### The Code Implementation

#### 1. The Sales Module (Pure & Dumb)
It only knows how to create a "Pending" order. It does *not* handle payments.

```javascript
// src/modules/sales/sales.public.js (Public Interface)
const salesService = require('./sales.service');

module.exports = {
    createPendingOrder: async (data, client) => {
        // Creates order with status 'PENDING_PAYMENT'
        return await salesService.create(data, client); 
    },
    confirmOrder: async (orderId, client) => {
        // Updates status to 'COMPLETED'
        return await salesService.updateStatus(orderId, 'COMPLETED', client);
    },
    cancelOrder: async (orderId, client) => {
        return await salesService.updateStatus(orderId, 'CANCELLED', client);
    }
};
```

#### 2. The Payment Module (Pure & Dumb)
It only knows how to process a transaction.

```javascript
// src/modules/payments/payments.public.js

module.exports = {
    processPayment: async (amount, method, client) => {
        // Contacts Stripe/Razorpay or logs Cash
        // Returns { success: true, transactionId: '...' }
        return await paymentService.charge(amount, method, client);
    }
};
```

#### 3. The Orchestrator (The Smart Logic)
This is where the business logic for "Selling" actually lives.

**`src/workflows/checkout.workflow.js`**

```javascript
const db = require('../shared/database');
const salesModule = require('../modules/sales/sales.public');
const paymentModule = require('../modules/payments/payments.public');
const { eventBus, EVENTS } = require('../shared/event-bus');

async function executeCheckout(tenantId, userId, checkoutData) {
    const client = await db.connect();
    
    try {
        await client.query('BEGIN'); // Start Distributed Transaction

        // STEP 1: Ask Sales Module to draft the order
        // The Sales module doesn't know payment is about to happen.
        const order = await salesModule.createPendingOrder({
            tenantId,
            userId,
            items: checkoutData.items,
            total: checkoutData.total
        }, client);

        // STEP 2: Ask Payment Module to charge money
        // The Payment module doesn't know what items were bought.
        const payment = await paymentModule.processPayment(
            checkoutData.total, 
            checkoutData.paymentMethod, 
            client
        );

        if (!payment.success) {
            throw new Error("Payment Rejected");
        }

        // STEP 3: Confirm the Order
        await salesModule.confirmOrder(order.id, client);
        
        // STEP 4: Link Payment to Order (in the join/link table or via event)
        // Ideally, emit an event here so the Ledger knows.
        
        await client.query('COMMIT');

        // STEP 5: Async Side Effects (Inventory)
        // Now we can safely tell Inventory to deduct stock
        eventBus.emit(EVENTS.ORDER_CREATED, { 
            orderId: order.id, 
            items: checkoutData.items 
        });

        return { success: true, orderId: order.id };

    } catch (error) {
        await client.query('ROLLBACK');
        // If payment failed, the Pending Order is effectively deleted (rolled back)
        console.error("Checkout Failed:", error);
        throw error;
    } finally {
        client.release();
    }
}

module.exports = { executeCheckout };
```

### Why this is Architecturally Superior

1.  **Strict Isolation:**
    *   Open `src/modules/sales/package.json` (or just check imports). You will see **zero** references to "payments".
    *   Open `src/modules/payments`. You will see **zero** references to "orders".
    *   You can now swap out the Payment module entirely without touching a single line of code in the Sales module.

2.  **Transaction Safety:**
    *   By passing the database `client` (connection) to both modules, the Orchestrator ensures that if the payment fails, the database `ROLLBACK` removes the pending order automatically. You don't end up with "Ghost Orders" in your database.

3.  **Refactoring Capability:**
    *   If you later move to Microservices, you only change the **Orchestrator**.
    *   Instead of `await paymentModule.processPayment()`, the Orchestrator changes to `await axios.post('http://payment-service/...')`.
    *   The core logic inside Sales and Payments remains untouched.

### When to use what?

| Pattern | Communication | Use Case | Example |
| :--- | :--- | :--- | :--- |
| **Event-Driven** | Async (Fire & Forget) | Side effects, Analytics, Logging | `Sales` -> `Inventory` |
| **Orchestrator** | Sync (Request/Response) | Critical dependent flows | `Checkout` -> `Sales` + `Payment` |
| **Direct Import** | Sync (Tight Coupling) | Helper functions | `Sales Service` -> `Sales Utils` |

For your "Point of Sale" app, the **Orchestrator** is the correct pattern for the Checkout flow.

This is the industry-standard approach for SaaS applications (like Slack, Discord, or Notion). You are effectively separating **Authentication** (Who are you?) from **Authorization** (Which store do you belong to and what can you do there?).

### 2. The Authentication Flow

This architecture changes how you issue JWTs (JSON Web Tokens). You cannot just issue a token for "User John". You must issue a token for "User John **acting inside** Store A".

#### Step 1: Login (Authentication)
*   **Input:** Email, Password.
*   **Action:** Verify against `auth.identities`.
*   **Response:** Do **not** issue a full access token yet. Return a list of available stores.

```json

  "token": "temporary-identity-token", // Valid for 5 mins, allows selecting a store
  "identity": { "id": "user-123", "name": "John Doe" },
  "availableStores": [
    { "id": "store-A", "name": "John's Pizza", "role": "OWNER" },
    { "id": "store-B", "name": "Downtown Cafe", "role": "CASHIER" }
  ]
}
```

#### Step 2: Session Selection (Switching Context)
*   **Input:** `store_id` (User clicks on "Downtown Cafe").
*   **Action:** Verify that `user-123` is a member of `store-B`.
*   **Response:** Issue the **Real Access Token**.

**The JWT Payload:**
```json
{
  "sub": "user-123",       // Identity ID
  "tenantId": "store-B",   // Context
  "role": "CASHIER",       // Permissions
  "exp": 1712345678
}
```

### 3. Middleware Implementation (Express)

This makes your backend secure by default. Every request checks the role inside the token.

```javascript
// src/shared/middleware/auth.middleware.js

const verifyToken = (req, res, next) => {
    const token = req.headers.authorization?.split(' ')[1];
    if (!token) return res.status(401).send('Access Denied');

    try {
        const decoded = jwt.verify(token, process.env.JWT_SECRET);
        
        // Attach context to the request object
        req.user = {
            id: decoded.sub,
            tenantId: decoded.tenantId,
            role: decoded.role
        };
        next();
    } catch (err) {
        res.status(400).send('Invalid Token');
    }
};

const requireRole = (allowedRoles) => {
    return (req, res, next) => {
        // req.user is already populated by verifyToken
        if (!allowedRoles.includes(req.user.role)) {
            return res.status(403).json({ 
                error: "Forbidden", 
                message: `You need to be a ${allowedRoles.join(' or ')} to perform this action.` 
            });
        }
        next();
    };
};

module.exports = { verifyToken, requireRole };
```

### 3. The "Provisioning" Workflow
This is a specialized workflow. It does two things at once:
1.  Creates a **Tenant** (Store).
2.  Creates an **Identity** (The Store Owner).
3.  Links them as the **OWNER**.

Create a new controller: **`src/modules/admin/admin.controller.js`**

```javascript
const bcrypt = require('bcryptjs');
const db = require('../../shared/database'); // wrapper around prisma
const { PrismaClient } = require('@prisma/client');
const prisma = new PrismaClient();

async function provisionStore(req, res) {
    const { 
        storeName, 
        storeSlug, 
        ownerEmail, 
        ownerPassword, 
        ownerFirstName 
    } = req.body;

    try {
        // We use a Transaction to ensure we don't get half-created stores
        const result = await prisma.$transaction(async (tx) => {
            
            // 1. Check if Identity exists, or Create new one
            let ownerIdentity = await tx.identity.findUnique({
                where: { email: ownerEmail }
            });

            if (!ownerIdentity) {
                const hashedPassword = await bcrypt.hash(ownerPassword, 10);
                ownerIdentity = await tx.identity.create({
                    data: {
                        email: ownerEmail,
                        passwordHash: hashedPassword,
                        firstName: ownerFirstName,
                        systemRole: 'USER' // They are a User in the system, Owner in the store
                    }
                });
            }

            // 2. Create the Store
            const newStore = await tx.store.create({
                data: {
                    name: storeName,
                    slug: storeSlug,
                    ownerId: ownerIdentity.id
                }
            });

            // 3. Assign Role: Make them the OWNER of this new store
            await tx.storeMember.create({
                data: {
                    storeId: newStore.id,
                    identityId: ownerIdentity.id,
                    role: 'OWNER'
                }
            });

            return { store: newStore, owner: ownerIdentity };
        });

        res.json({
            success: true,
            message: "Store provisioned successfully",
            data: {
                storeId: result.store.id,
                storeName: result.store.name,
                ownerEmail: result.owner.email
            }
        });

    } catch (error) {
        console.error(error);
        res.status(500).json({ error: "Provisioning failed", details: error.message });
    }
}

module.exports = { provisionStore };
```

```javascript
const { PrismaClient } = require('@prisma/client');
const prisma = new PrismaClient(); // Or import your shared db instance
const authPublic = require('../modules/auth/auth.public');
const tenantsPublic = require('../modules/tenants/tenants.public');

/**
 * Orchestrates the creation of a new Tenant and its Owner.
 * Transactional: If store creation fails, the user isn't created (or vice versa).
 */
async function provisionTenantWorkflow(input) {
    const { 
        storeName, 
        storeSlug, 
        ownerEmail, 
        ownerPassword, 
        ownerFirstName 
    } = input;

    // Start a Global Transaction
    return await prisma.$transaction(async (tx) => {
        
        // STEP 1: Auth Module - Ensure Identity Exists
        // We pass 'tx' so this happens inside the transaction
        const identity = await authPublic.ensureIdentity({
            email: ownerEmail,
            password: ownerPassword, // Auth module will hash this
            firstName: ownerFirstName,
            systemRole: 'USER' // Platform Admin is creating a Standard User
        }, tx);

        // STEP 2: Tenants Module - Create the Store
        const store = await tenantsPublic.createStore({
            name: storeName,
            slug: storeSlug,
            ownerId: identity.id
        }, tx);

        // STEP 3: Tenants Module - Assign 'OWNER' Role
        await tenantsPublic.assignRole({
            storeId: store.id,
            identityId: identity.id,
            role: 'OWNER'
        }, tx);

        // Return a clean DTO (Data Transfer Object)
        return {
            storeId: store.id,
            storeName: store.name,
            ownerEmail: identity.email,
            ownerId: identity.id
        };
    });
}

module.exports = { provisionTenantWorkflow };
```

if I want to include versioning, how would I organize my module structure

Versioning in a **Modular Monolith** is best handled **inside the module** rather than at the root level.

If you create `src/v1/...` and `src/v2/...` at the root, you end up duplicating **all** modules even if only one of them changed.

Instead, use the **"Internal Versioning"** strategy. Only the modules that *change* get a V2 folder, while the rest stay as they are.

Here is the directory structure and implementation guide using ESM.

### 1. The Directory Structure

Move your Controllers and Routes into versioned folders (`v1`, `v2`). Keep your **Services** and **Models** shared (usually representing the "latest" state).

```text
src/
  modules/
    sales/
      ├── sales.service.js       <-- Core Logic (Always Latest)
      ├── sales.model.js         <-- Prisma/DB helpers
      │
      ├── v1/                    <-- Legacy API Support
      │   ├── sales.controller.js
      │   ├── sales.routes.js
      │   └── sales.dto.js       <-- Data validation for V1
      │
      ├── v2/                    <-- The New Standard
      │   ├── sales.controller.js
      │   ├── sales.routes.js
      │   └── sales.dto.js
      │
      └── index.js               <-- Module Entry Point
```

---

### 2. The Strategy: "The Adapter Pattern"

**The Golden Rule:** Do **not** copy-paste `sales.service.js` into `v1` and `v2`.
Your Database Schema is moving forward. Your Service should reflect the current Schema.

*   **V2 Controller:** Calls the Service directly (1:1 mapping).
*   **V1 Controller:** Acts as an **Adapter**. It accepts the old data shape, massages it into the new shape, calls the Service, and then massages the response back to the old shape.

---

1.  **Routes (`routes.js`)** are the **Host/Hostess**.
    *   They stand at the door.
    *   They look at your request ("I want a table for two" vs "I want to pick up an order").
    *   They guide you to the right waiter.
    *   *They do not cook the food, and they do not serve the food.*

2.  **Controllers (`controller.js`)** are the **Waiters**.
    *   They come to your table (the endpoint).
    *   They take your order (read `req.body`).
    *   They check if your order makes sense ("You can't order a hamburger without a bun").
    *   They tell the Kitchen/Chef (the **Service Layer**) to cook it.
    *   When the food is ready, they bring it to you (send `res.json`).

---

### 1. The Routes File (`routes.js`)
**Role:** The Traffic Cop / Table of Contents.
This file should contain **zero logic**. Its only job is to map a URL + HTTP Method to a specific Controller function and apply Middleware (security).

**Example: `src/modules/products/products.routes.js`**
```javascript
import { Router } from 'express';
import * as productController from './products.controller.js';
import { verifyToken, requireRole } from '../../shared/middleware/auth.middleware.js';

const router = Router();

// 1. URL Definition: POST /
// 2. Security: verifyToken
// 3. Destination: productController.createProduct
router.post('/', verifyToken, productController.createProduct);

// 1. URL Definition: DELETE /:id
// 2. Security: verifyToken AND requireRole
// 3. Destination: productController.deleteProduct
router.delete('/:id', 
    verifyToken, 
    requireRole(['ADMIN']), 
    productController.deleteProduct
);

export default router;
```

---

### 2. The Controller File (`controller.js`)
**Role:** The HTTP Handler.
This file interacts with the "Web" part of your app. It knows about `req` (Request) and `res` (Response). It should **never** talk to the database directly (that's the Service's job).

**Responsibilities:**
1.  **Extract Data:** Get `id` from URL or `name` from the Body.
2.  **Validate Input:** "Is the price greater than 0?"
3.  **Call Service:** "Hey Service, create this product."
4.  **Send Response:** "201 Created" or "400 Bad Request".

**Example: `src/modules/products/products.controller.js`**
```javascript
import * as productService from './products.service.js';

export const createProduct = async (req, res) => {
    try {
        // 1. Extract & Validate
        const { name, price } = req.body;
        
        if (!name || price < 0) {
            // Controller decides strictly HTTP concerns (Status 400)
            return res.status(400).json({ error: "Invalid input" });
        }

        // 2. Call Business Logic (The Service)
        // Notice: The controller doesn't know about SQL or Prisma
        const newProduct = await productService.create({ 
            tenantId: req.user.tenantId, // Taken from middleware
            name, 
            price 
        });

        // 3. Send Response
        res.status(201).json({ 
            success: true, 
            data: newProduct 
        });

    } catch (error) {
        // Controller handles the HTTP error code
        console.error(error);
        res.status(500).json({ error: "Internal Server Error" });
    }
};
```

---

### Summary: The Difference

| Feature | **Routes (`routes.js`)** | **Controllers (`controller.js`)** |
| :--- | :--- | :--- |
| **Main Job** | Pointing to the right place. | Handling the input/output. |
| **Knows about Database?** | No. | No (Should delegate to Service). |
| **Knows about `req` / `res`?** | No (It just passes them). | **Yes** (This is its main tool). |
| **Contains Logic?** | **No** (Only middleware logic). | **Yes** (Input validation & flow control). |
| **Analogy** | The Menu / Receptionist. | The Waiter. |

### Why separate them?
If you put your controller logic *inside* your route file, your route file becomes 1,000 lines long. By separating them:
1.  **Readability:** You can open `routes.js` and see a clean list of every feature your API supports in 5 seconds.
2.  **Organization:** Logic stays with logic, configuration stays with configuration.

```javascript
/**
 * Step 1: Login
 * Input: email, password
 * Output: { tempToken, availableStores: [...] }
 */
export const login = async (req, res) => {
    try {
        const { email, password } = req.body;

        const result = await authService.authenticateIdentity(email, password);
        
        if (!result) {
            return res.status(401).json({ error: "Invalid credentials" });
        }

        res.json({
            success: true,
            message: "Identity verified. Please select a store.",
            tempToken: result.tempToken, // Short-lived (5 mins)
            availableStores: result.stores // [{ id: 'store-A', name: "Joe's Pizza", role: "OWNER" }]
        });
    } catch (error) {
        res.status(500).json({ error: error.message });
    }
};

/**
 * Step 2: Select Store
 * Input: tempToken (in headers), storeId
 * Output: { accessToken, store: { ... } }
 */
export const selectStore = async (req, res) => {
    try {
        const { storeId } = req.body;
        // In a real app, you'd extract the tempToken from Authorization header
        // For simplicity here, assuming it's passed or handled by middleware
        const tempToken = req.headers.authorization?.split(' ')[1];

        // Service verifies tempToken and checks if user is actually in storeId
        const result = await authService.issueStoreToken(tempToken, storeId);

        res.json({
            success: true,
            accessToken: result.accessToken, // Long-lived (1 day), contains tenantId
            user: result.user,
            store: result.store
        });
    } catch (error) {
        res.status(403).json({ error: "Access to this store denied" });
    }
};

/**
 * Get Profile
 */
export const getMe = async (req, res) => {
    // req.user is populated by verifyToken middleware
    res.json({
        id: req.user.id,
        email: req.user.email,
        currentStore: req.user.tenantId,
        role: req.user.role
    });
};
```

so will the admin routes have its own login

**No, you do not need a separate login route.**

Having a separate `/admin/login` is an anti-pattern. It creates code duplication and confuses users who might be both a Store Owner *and* a Platform Admin.

Instead, you use the **Single Sign-On (SSO)** approach we already built. The **Identity** is the same; only the **Permissions** (Claims) inside the token change.

Here is how to handle the **Platform Admin** flow using your existing `/auth/login`.

---

### 1. The Concept: "Identity Token" vs. "Store Token"

*   **Standard User Flow:**
    1.  Login → Get **Temp Identity Token**.
    2.  Select Store → Exchange for **Store Token** (contains `tenantId`).
    3.  Access Store Data.

*   **Platform Admin Flow:**
    1.  Login → Get **Full Identity Token** (contains `systemRole: 'PLATFORM_ADMIN'`).
    2.  **Branch A:** Call `/admin/provision` immediately (using Identity Token).
    3.  **Branch B:** Select a Store → Exchange for **Store Token** (to act as a user inside a store).

---

### 2. Update the Login Service

We need to tweak `auth.service.js` so that if the user is a `PLATFORM_ADMIN`, the token we issue is **powerful enough** to access Admin routes immediately, without selecting a store.

**`src/modules/auth/auth.service.js`**

```javascript
import jwt from 'jsonwebtoken';
// ... imports

export const authenticateIdentity = async (email, password) => {
    // 1. Verify Credentials (Standard)
    const identity = await prisma.identity.findUnique({ where: { email } });
    if (!identity) return null;

    const valid = await bcrypt.compare(password, identity.passwordHash);
    if (!valid) return null;

    // 2. Determine Token Power
    // If they are PLATFORM_ADMIN, this token is valid for /admin routes immediately.
    // If they are USER, this is just a temporary token for store selection.
    const isSuperAdmin = identity.systemRole === 'PLATFORM_ADMIN';

    const tokenPayload = {
        sub: identity.id,
        email: identity.email,
        systemRole: identity.systemRole, // 'USER' or 'PLATFORM_ADMIN'
        // Note: No 'tenantId' here yet!
    };

    const token = jwt.sign(
        tokenPayload, 
        process.env.JWT_SECRET, 
        { expiresIn: isSuperAdmin ? '1d' : '15m' } // Admins get a real session immediately
    );

    // 3. Get Stores (Standard)
    const members = await prisma.storeMember.findMany({
        where: { identityId: identity.id },
        include: { store: true }
    });

    return {
        token,
        isSuperAdmin, // Flag for the Frontend to know what UI to show
        stores: members.map(m => ({
            id: m.store.id,
            name: m.store.name,
            role: m.role
        }))
    };
};
```

where can i place a reusable function for formatting error responses

In a Modular Monolith, shared logic belongs in the **`src/shared`** directory.

For error handling, you shouldn't just create a "formatter function." Instead, you should implement a **Centralized Error Handling Strategy**. This removes `try/catch` blocks from your controllers and ensures every error (Auth, Database, Validation) looks exactly the same to the frontend.

Here is the professional way to structure this.

### 1. The Directory Structure
Create a `utils` folder inside shared.

```text
src/
  shared/
    middleware/
      error.middleware.js  <-- The Global Error Catcher
    utils/
      AppError.js          <-- The Custom Error Class
      catchAsync.js        <-- The Wrapper (Removes try/catch)
```

---

### 2. The Custom Error Class (`AppError.js`)
This class extends the native JS `Error`. It allows you to throw errors with a status code anywhere in your app.

**`src/shared/utils/AppError.js`**
```javascript
export class AppError extends Error {
  constructor(message, statusCode) {
    super(message);

    this.statusCode = statusCode;
    // If 4xx, status is 'fail'. If 5xx, status is 'error'
    this.status = `${statusCode}`.startsWith('4') ? 'fail' : 'error';
    this.isOperational = true; // Distinguish known errors from bugs

    Error.captureStackTrace(this, this.constructor);
  }
}
```

---

### 3. The Global Middleware (`error.middleware.js`)
This is the **"Reusable Formatter"** you asked for. It sits at the very end of your Express app and formats *every* error uniformly.

**`src/shared/middleware/error.middleware.js`**
```javascript
export const globalErrorHandler = (err, req, res, next) => {
  err.statusCode = err.statusCode || 500;
  err.status = err.status || 'error';

  // Development: Send full stack trace
  if (process.env.NODE_ENV === 'development') {
    return res.status(err.statusCode).json({
      status: err.status,
      error: err,
      message: err.message,
      stack: err.stack,
    });
  }

  // Production: Don't leak code details to the client
  // 1. Trusted Error (We threw it manually using AppError)
  if (err.isOperational) {
    return res.status(err.statusCode).json({
      status: err.status,
      message: err.message,
    });
  }

  // 2. Programming Error (Bug, Prisma Crash, etc.)
  console.error('ERROR 💥', err); // Log it for yourself
  return res.status(500).json({
    status: 'error',
    message: 'Something went very wrong!',
  });
};
```

---

### 4. The `catchAsync` Wrapper (Optional but Recommended)
This simple utility removes the need to write `try { ... } catch (e) { next(e) }` in every single controller.

**`src/shared/utils/catchAsync.js`**
```javascript
/**
 * Wraps async functions and automatically passes errors to Next()
 */
export const catchAsync = (fn) => {
  return (req, res, next) => {
    fn(req, res, next).catch(next);
  };
};
```

---

### 5. How to use it in your Controllers

Look how clean your `Admin Controller` becomes. No `try/catch`, no manual JSON formatting.

**`src/modules/admin/admin.controller.js`**
```javascript

import { catchAsync } from '../../shared/utils/catchAsync.js';
import { provisionTenantWorkflow } from '../../workflows/provisioning.workflow.js';

    const { storeName, ownerEmail } = req.body;

    // 1. Validate Input - Throw AppError
    if (!storeName || !ownerEmail) {
        throw new AppError('Store name and Owner email are required', 400);
    }

    // 2. Run Workflow
    // If Prisma throws an error here, catchAsync catches it automatically
    const result = await provisionTenantWorkflow(req.body);

    res.status(201).json({
        success: true,
        data: result
    });
});
```

### 6. Wiring it up in `app.js`

You must mount the global error handler **after** all routes.

**`src/app.js`**
```javascript
import express from 'express';
import { globalErrorHandler } from './shared/middleware/error.middleware.js';
import { AppError } from './shared/utils/AppError.js';
// ... imports

const app = express();

// ... routes ...

// Handle 404 (Unknown Routes)
app.all('*', (req, res, next) => {
  next(new AppError(`Can't find ${req.originalUrl} on this server!`, 404));
});

// GLOBAL ERROR HANDLER (Must be last)

export default app;
```

### Summary
*   **Where?** `src/shared/utils/AppError.js` and `src/shared/middleware/error.middleware.js`.
*   **Why?** It enforces a standard JSON error structure `{ status: 'fail', message: '...' }` across all modules without duplicating code.

do you see any issues with this 
app.all("*", (req, res, next) => {
  next(new AppError(`Can't find ${req.originalUrl} on this server!`, 404));
});

The code snippet itself is **syntactically correct** and follows standard Express patterns.

However, there is **one critical integration requirement** regarding **where** you place it in your `app.js` file.

### 1. The Placement Issue (Critical)
This code matches **every** request (`*`).
*   If you place it **before** your routes, it will block every request, and your API will always return 404.
*   If you place it **after** the Global Error Handler, it will never be reached.

It must sit exactly in this "sandwich":

```javascript
// 1. Middlewares (Body parser, CORS, etc.)
app.use(express.json());

// 2. Your Routes
app.use('/api/v1', v1Router); 

// 3. THIS 404 HANDLER (Catches anything not matched above)
app.all("*", (req, res, next) => {
  next(new AppError(`Can't find ${req.originalUrl} on this server!`, 404));
});

// 4. Global Error Handler (Catches the error passed to 'next')
app.use(globalErrorHandler);
```

### 2. Dependency on `globalErrorHandler`
This code uses `next(new AppError(...))`.
This **only works** if you have actually registered the `globalErrorHandler` (the one with 4 arguments: `err, req, res, next`) immediately after it.

If you forget step #4 in the specific order above, the Express default HTML error page will be shown instead of your nice JSON response.

### 3. Minor Edge Case: Method Not Allowed (405)
Strictly speaking, if a user sends a `POST` request to a route that only supports `GET`, standard HTTP semantics suggest a **405 Method Not Allowed**.

Your code will return **404 Not Found**.
*   **Is this a problem?** For 99% of APIs, **No**. Returning 404 is often considered more secure because it doesn't reveal to attackers that a route exists but they are just using the wrong method.
*   **Verdict:** Keep it as 404. It's standard, safer, and easier.

### Summary
The code is fine, just make sure it is the **last route** defined in your `app.js` and the **Global Error Handler** comes immediately after it.

I am getting this error though 
PathError [TypeError]: Missing parameter name at index 1: *; visit https://git.new/pathToRegexpError for info


This error happens because you are likely using **Express v5** (beta) or a newer version of the underlying `path-to-regexp` library, where the simple `"*"` wildcard syntax has been changed or removed.

In newer versions, `*` is a reserved character for parameters and cannot be used as a standalone wildcard anymore.

### The Fix

You have two easy options to fix this:

#### Option 1: Use Regex (Most Robust)
Pass a Regular Expression instead of a string. This works in almost all versions of Express.

```javascript
// Change "*" to /(.*)/
app.all(/(.*)/, (req, res, next) => {
  next(new AppError(`Can't find ${req.originalUrl} on this server!`, 404));
});
```

#### Option 2: Use `(.*)` (String Syntax)
If you want to keep using a string, you must explicitly define the capture group.

```javascript
// Change "*" to "(.*)"
app.all("(.*)", (req, res, next) => {
  next(new AppError(`Can't find ${req.originalUrl} on this server!`, 404));
});
```

### Why did this break?
In the older Express v4, `*` was a special shortcut. In the new router logic (Express 5+), they strictly follow standard URL parameter rules where `*` implies "repeat the previous parameter," so a standalone `*` is considered a syntax error (missing parameter name).

Where do i define api routes for complex workflows that orchestrate different modules

This is a critical architectural decision.

Since you are building a **Modular Monolith**, you have **two main places** where complex, cross-module workflows (like "Checkout") can live.

### Option 1: The "Primary Module" Owns It (Recommended)
Usually, a workflow clearly "belongs" to one domain more than others.
*   **Checkout:** While it involves Payments and Inventory, it is fundamentally a **Sales** process. The user is "buying something."
*   **Onboarding:** Involves Auth and Tenants, but is fundamentally an **Admin** process.

**Decision:** Place the route in the **Sales Module**.
*   **Route:** `POST /api/v1/sales/checkout`
*   **Controller:** `src/modules/sales/sales.controller.js`
*   **Logic:** The controller calls `checkoutWorkflow.execute(...)` (which imports other modules).

### Option 2: A Dedicated "Gateway" Module (Advanced)
If you have a workflow that truly has **no owner** (e.g., a "Global Search" that searches Products, Customers, and Orders simultaneously), create a **Gateway** or **BFF (Backend for Frontend)** module.

**Structure:**
```text
src/
  modules/
    gateway/
      gateway.routes.js     <-- /api/v1/gateway/search
      gateway.controller.js
```

### Implementing Zod Validation for Complex Workflows

Since Zod schemas can get large, do **not** put them in the controller. Create a `*.schema.js` file next to your controller.

**1. Define the Schema (`src/modules/sales/sales.schema.js`)**

```javascript
import { z } from 'zod';

export const checkoutSchema = z.object({
  body: z.object({
    // Complex validation logic here
    items: z.array(z.object({
      productId: z.string().uuid(),
      quantity: z.number().int().positive(),
    })).min(1, "Must buy at least one item"),
    
    paymentMethod: z.enum(['CASH', 'CARD', 'UPI']),
    
    // Optional: Coupon Code
    couponCode: z.string().optional(),
  }),
});
```

**2. Create a Validation Middleware (`src/shared/middleware/validate.middleware.js`)**
This reusable middleware checks `req.body`, `req.query`, and `req.params` against your Zod schema.

```javascript
import { z } from 'zod';
import { AppError } from '../utils/AppError.js';

export const validate = (schema) => (req, res, next) => {
  try {
    // Validate request against schema
    schema.parse({
      body: req.body,
      query: req.query,
      params: req.params,
    });
    next();
  } catch (err) {
    if (err instanceof z.ZodError) {
      // Format Zod errors nicely
      const errorMessage = err.errors.map((e) => e.message).join(', ');
      return next(new AppError(`Validation Error: ${errorMessage}`, 400));
    }
    next(err);
  }
```

**Yes, having the database auto-generate the Session ID is perfectly fine and is the standard approach.**

There is rarely a need for the application to generate the ID manually unless you are doing complex optimistic UI updates or specific cryptographic binding before the database touch.

Here is the standard flow with your current schema:
1.  **User logs in.**
2.  **App:** Inserts a row into `Session` (DB generates ID: `uuid-1`).
3.  **App:** Takes `uuid-1` and puts it inside the JWT Access Token as a claim (e.g., `sid: "uuid-1"`).
4.  **App:** Sends the JWT to the frontend.

This allows you to revoke a specific session later by checking the `sid` in the token against the database.

---

### ⚡ Architectural Optimization: Merge Session & RefreshToken

I noticed you have a **1-to-1 relationship** between `Session` and `RefreshToken`.
In 99% of implementations, **the Refresh Token IS the Session.**

Separating them creates unnecessary complexity:
1.  **Performance:** You have to do a `JOIN` to check if a token is valid.
2.  **Storage:** You are storing two IDs (Session ID + Refresh Token ID) for the same concept.

**Recommended Schema Refactor:**
Merge them into a single table. This makes your Auth module faster and simpler.

```prisma
model Session {
  id            String    @id @default(uuid()) // This is your "Session ID"
  identity_id   String
  
  // Security: Store the HASH of the refresh token, not the raw token
  // If your DB is leaked, attackers can't hijack sessions.
  token_hash    String    
  
  user_agent    String?
  ip_address    String?   // Good for audit trails
  
  created_at    DateTime  @default(now())
  expires_at    DateTime  // When the Refresh Token dies
  
  // Relations
  identity      Identity  @relation(fields: [identity_id], references: [id], onDelete: Cascade)

  @@map("sessions")
  @@schema("auth")
}
```

### 🔒 Security Best Practice: The "Opaque" Token

When you implement the logic in `auth.service.js`, follow this pattern:

1.  **Generate a Random String** (The Refresh Token).
    *   `const rawRefreshToken = crypto.randomBytes(32).toString('hex');`
2.  **Hash It.**
    *   `const tokenHash = await argon2.hash(rawRefreshToken);`
3.  **Save to DB.**
    *   Save `tokenHash` to the `sessions` table.
4.  **Send to User.**
    *   Send `rawRefreshToken` to the user (via HttpOnly Cookie).

**Why?**
If you store the raw token in the database (as in your current schema), and someone hacks your database, **they can log in as anyone** because they have all the active refresh tokens. If you store the hash, the leaked data is useless to the attacker.

To get the client's IP address in an Express app, you need to handle the fact that your app will likely sit behind a proxy (like Nginx, AWS Load Balancer, or Cloudflare) in production.

### 1. The Utility Function (`src/shared/utils/getIp.js`)

Create a helper because the IP can be hidden in different headers depending on your hosting provider.

```javascript
export const getClientIp = (req) => {
  // 1. Try standard proxy headers (Nginx, AWS, Cloudflare)
  const forwarded = req.headers['x-forwarded-for'];
  
  // x-forwarded-for can be a list: "client_ip, proxy1_ip, proxy2_ip"
  // We want the FIRST one.

    req.connection?.remoteAddress ||
    null
  );
};
```

I want to add a middleware that parses the token and populates an auth context on the req. Would this be under the shared middleware folder or under the auth module folder.

This is a subtle architectural decision.

If you place the middleware in the **Auth Module** (`src/modules/auth/middleware/auth.middleware.js`), it makes sense because **Auth knows how to parse Auth**.

However, if you place it in **Shared** (`src/shared/middleware/auth.middleware.js`), it makes sense because **Every module needs it**.

**Recommendation: Place it in `src/shared/middleware/auth.middleware.js`**.

Here is why:
1.  **Cross-Cutting Concern:** Authentication is not just an "Auth Module" feature. The **Inventory Module** needs to know `req.user.role`. The **Sales Module** needs `req.user.tenantId`.
2.  **Circular Dependencies:** If `Sales` imports `Auth Middleware`, and `Auth` imports `Sales Service` (e.g., to create a default sale on signup), you get a circular dependency. Putting it in `Shared` avoids this.

### The Implementation (`src/shared/middleware/auth.middleware.js`)

Here is the robust, production-ready middleware.

```javascript
import jwt from 'jsonwebtoken';
import { AppError } from '../utils/AppError.js';
import { catchAsync } from '../utils/catchAsync.js';
import { PrismaClient } from '@prisma/client';

const prisma = new PrismaClient(); // Or import shared instance

export const protect = catchAsync(async (req, res, next) => {
  // 1. Get token
  let token;
  if (
    req.headers.authorization &&
    req.headers.authorization.startsWith('Bearer')
  ) {
    token = req.headers.authorization.split(' ')[1];
  }

  if (!token) {
    return next(new AppError('You are not logged in!', 401));
  }

  // 2. Verify token
  // This throws JsonWebTokenError if invalid (caught by catchAsync)
  const decoded = jwt.verify(token, process.env.JWT_SECRET);

  // 3. Check if user still exists
  // (Optional: Only if you allow deleting users mid-session)
  const currentUser = await prisma.identity.findUnique({
    where: { id: decoded.id },
  });

  if (!currentUser) {
    return next(
      new AppError('The user belonging to this token no longer exists.', 401)
    );
  }

  // 4. Check if user changed password after the token was issued
  // (Optional: Compare decoded.iat < currentUser.passwordChangedAt)

  // 5. GRANT ACCESS TO PROTECTED ROUTE
  req.user = currentUser;
  req.session = { id: decoded.sessionId }; // Store session context too!
  next();
});
```

```javascript
            data: {
                storeId,
                identityId: identity.id,

            }
        });

    });
```

i am using jose library to sign my tokens. Does it allow me to add custom payload items in my jwt?

**Yes, absolutely.**

The `jose` library is excellent for this. Unlike the older `jsonwebtoken` library where you pass a big object, `jose` uses a **Builder Pattern**.

You pass your **custom claims** (like `tenantId`, `role`, `sessionId`) directly into the `SignJWT` constructor, and you set **standard claims** (like `exp`, `sub`) using chainable methods.

Here is how to implement your **Token Service** using `jose`.

### 1. Signing the Token (Adding Custom Data)

Create this utility in `src/shared/utils/token.utils.js`.

**Crucial Note:** `jose` requires the secret key to be a `Uint8Array`, not a string. We use `new TextEncoder().encode()` to fix this.

```javascript
import { SignJWT } from 'jose';

export const signToken = async (payload, secret, expiresIn) => {
  const secretKey = new TextEncoder().encode(secret);

  // 1. Separate Standard Claims from Custom Claims
  const { sub, ...customClaims } = payload;

  return await new SignJWT(customClaims) // <--- YOUR CUSTOM DATA GOES HERE
    .setProtectedHeader({ alg: 'HS256' })
    .setSubject(sub) // Standard 'sub' (Subject/User ID)
    .setIssuedAt()
    .setExpirationTime(expiresIn) // e.g., '15m' or '7d'
    .sign(secretKey);
};
```

Here is how you would use it to generate the **Store Access Token** we discussed (containing the Session ID and Tenant ID).

A **Refresh Token** represents the **"Identity Session"** (the fact that "I am John").

You need to split the logic **after** the session creation but **before** the response.

### 1. The Refactored Controller (`auth.controller.js`)

```javascript
    identity,
    userAgent,
    ipAddress,
  );

  // This ensures that even if they close the tab, they stay "Identified"
  res.cookie("refreshToken", refreshToken, {

    secure: process.env.NODE_ENV === "production",
    sameSite: "strict",
    path: "/api/auth/refresh",

      sub: identity.id,
      sid: session.id,

    return res.json({
      success: true,
      token: adminAccessToken,
      role: 'PLATFORM_ADMIN',
      redirectTo: '/admin/dashboard' // Frontend hint
    });
  }

  // SCENARIO B: Standard User
  // We don't know WHICH store they want to work in yet.
  // Issue a "Selection Token" (Scope: identity_only).
  const tempToken = await signToken({
    sub: identity.id,
    sid: session.id,
    scope: 'STORE_SELECTION' // Crucial: This token CANNOT access store data
```

```javascript
  const { refreshToken } = req.cookies;

  const session = await authService.verifySession(refreshToken);

  if (session.identity.system_role === 'PLATFORM_ADMIN') {
     const token = await signToken({ ...adminClaims });
     return res.json({ token });
  }

  // If they are User, we have a problem:
  // We don't know which store they were working in!
  
  // SOLUTION: The Frontend must send the 'tenantId' in the BODY of the refresh request
  // if they are currently inside a store dashboard.
  const { currentStoreId } = req.body;

     // Verify they still belong to this store

        sub: session.identity.id,
        sid: session.id,

  res.json({ token: tempToken, requireSelection: true });
```

should app errors always be thrown from the controller?

**No.**

Throwing errors from the **Service Layer** is not only acceptable, it is **encouraged**.

If you force all errors to be thrown in the Controller, you end up with "Leaky Abstractions." The Controller has to know too much about the business rules.

### Where to throw what?

| Error Type | **Where to throw?** | **Example** |
| :--- | :--- | :--- |
| **Validation (HTTP)** | **Controller / Middleware** | "Email is required", "Invalid JSON body" |
| **Business Logic** | **Service** | "Insufficient stock", "User already in store", "Invalid coupon" |
| **Data Integrity** | **Service / Repository** | "Record not found", "Duplicate email" |
| **Authentication** | **Middleware** | "Token expired", "Invalid signature" |

---

### Why Services should throw AppError

Imagine your **Inventory Service** has a method `deductStock`.
This logic is reusable. It might be called by:
1.  **Sales Controller** (User buys item).
2.  **Admin Controller** (Manager manually adjusts stock).
3.  **Background Job** (Subscription renewal).

If `deductStock` returns `false` or `null` on failure, **every caller** has to handle the error manually.

**Bad Pattern (Silent Failure):**
```javascript
// Service
const deductStock = async (itemId, qty) => {
  const item = await db.item.find(itemId);
  if (item.qty < qty) return false; // Controller must check this!
  // ...
};

// Controller
const result = await inventoryService.deductStock(id, 5);
if (!result) {
   throw new AppError("Out of stock", 400); // Duplicate logic in every controller
}
```

**Good Pattern (Throw Early):**
```javascript
// Service
import { AppError } from '../../shared/utils/AppError.js';

const deductStock = async (itemId, qty) => {
  const item = await db.item.find(itemId);
  
  if (!item) {
    throw new AppError("Item not found", 404);
  }
  
  if (item.qty < qty) {
    throw new AppError(`Insufficient stock. Only ${item.qty} available.`, 409);
  }
  
  // ... proceed ...
};

// Controller (Clean!)
// The catchAsync wrapper automatically catches the error from the service
await inventoryService.deductStock(id, 5);
res.json({ success: true });
```

### The Rule of Thumb
*   **Controller:** Handle "How the request arrived" (Parsers, Zod validation).
*   **Service:** Handle "What went wrong with the logic" (Business Rules).

Since `AppError` is a custom class, your Global Error Handler catches it regardless of where it was thrown.

Should service layer be aware of http status codes? like 404 and 403

This is one of the classic debates in backend architecture.

**The Short Answer:**
In a strict "Clean Architecture" sense, **No**. Your business logic should not know that it is running on a web server. It shouldn't care about HTTP 404 vs 403.

**The Pragmatic Answer (for your specific SaaS):**
**Yes, but abstract it.** Writing 50 different error classes (`UserNotFound`, `ProductNotFound`, `OrderNotFound`, `InsufficientFunds`) and mapping them manually in the controller is overkill and slows down development.

Here is the **Hybrid Approach** that gives you the best of both worlds: **Semantic Error Classes**.

---

### The Strategy: Semantic Wrappers
Instead of throwing `new AppError('msg', 404)` directly in your service (which hardcodes the number 404), you create specific error classes that *extend* `AppError`.

This keeps your Service reading like **English** (Business Logic) rather than **Protocol** (HTTP Codes).

#### 1. Define these in `src/shared/utils/errors.js`

```javascript
import { AppError } from './AppError.js';

// 404
export class NotFoundError extends AppError {
  constructor(message = 'Resource not found') {
    super(message, 404);
  }
}

// 403
export class ForbiddenError extends AppError {
  constructor(message = 'Access denied') {
    super(message, 403);
  }
}

// 401
export class UnauthorizedError extends AppError {
  constructor(message = 'Invalid credentials') {
    super(message, 401);
  }
}

// 400
export class BadRequestError extends AppError {
  constructor(message = 'Invalid input') {
    super(message, 400);
  }
}

// 409
export class ConflictError extends AppError {
  constructor(message = 'Resource already exists') {
    super(message, 409);
  }
}
```

#### 2. Usage in the Service Layer
Now your service looks like this. It is clean, readable, and technically "transport agnostic" because `NotFoundError` could theoretically be mapped to a gRPC code or CLI exit code later if you changed the implementation of the class.

**`src/modules/inventory/inventory.service.js`**

```javascript
import { NotFoundError, BadRequestError } from '../../shared/utils/errors.js';

export const deductStock = async (itemId, quantity) => {
  const item = await db.item.findUnique({ where: { id: itemId } });

  // 1. Semantic check: "Not Found"
  if (!item) {
    throw new NotFoundError(`Item with ID ${itemId} does not exist`);
  }

  // 2. Semantic check: "Bad Request" (Business Rule Violation)
  if (item.stock < quantity) {
    throw new BadRequestError(`Only ${item.stock} items remaining`);
  }

  // ... update logic
};
```

### Why is this better?

1.  **Readability:** `throw new NotFoundError()` is self-explanatory. `throw new AppError(..., 404)` forces the reader to remember what 404 means.
2.  **Refactoring:** If you decide later that "Insufficient Stock" should be a **422 Unprocessable Entity** instead of a **400 Bad Request**, you don't have to hunt through 50 service files. You just change the `BadRequestError` class or create a new `BusinessRuleError`.
3.  **Consistency:** You prevent developers (or yourself) from accidentally using 404 when you meant 400.

### When to break the rule?

If you are just prototyping or writing a very simple script, passing `404` directly is fine. But for a SaaS with multiple modules and potential for growth, **Semantic Error Classes** are the standard industry practice.

when calling the refresh endpoint I pass the refresh token and the db has the hashed token. I use argon2 for hashing the refresh token. How do i query the session table using the refresh token since the hashes dont match?

This is a classic authentication problem called **"Indexing Hashed Values"**.

You **cannot query** the database directly for the Refresh Token because argon2 produces a different hash (salt) every time, even for the same input. That's its security feature!

So, you have two options:

### Option 1: Store the Token ID (Recommended)
You split the Refresh Token into two parts: `TokenID` and `TokenSecret`.

1.  **Generate:**
    *   `TokenID` = `uuid()` (e.g., `user-session-123`)
    *   `TokenSecret` = `randomBytes(32)` (e.g., `abc...xyz`)
    *   `FullToken` = `TokenID.TokenSecret` (Send to User).
2.  **Hash:**
    *   Hash `TokenSecret` ONLY.
    *   Store `TokenID` (Plaintext) + `TokenSecretHash` (Argon2) in DB.
3.  **Query:**
    *   User sends `FullToken`. Split it by `.`.
    *   `SELECT * FROM sessions WHERE id = TokenID`.
    *   `argon2.verify(session.token_hash, TokenSecret)`.

**Pros:** Fast lookup (indexed by ID). Secure (Secret is hashed).
**Cons:** Requires slightly more logic in generation/parsing.

### Option 2: Use Fast Hash for Lookup (Simpler)
Use a deterministic hash (SHA-256) for lookup and Argon2 for verification.
*   **Store:** `lookup_hash` (SHA-256 of token) + `verify_hash` (Argon2 of token).
*   **Query:** `WHERE lookup_hash = SHA256(input_token)`.

**Pros:** Simple.
**Cons:** Double hashing. SHA-256 is fast (brute-forceable if token is short), but okay for high-entropy tokens.

### Recommendation: Option 1 (Token ID + Secret)

Here is how to implement it cleanly.

#### 1. Token Generation (`auth.service.js`)

```javascript
import { randomBytes } from 'crypto';
import argon2 from 'argon2';

export const createRefreshToken = async () => {
  // 1. Generate Parts
  const tokenId = crypto.randomUUID(); // The lookup key
  const tokenSecret = randomBytes(32).toString('hex'); // The secure part

  // 2. Hash the Secret
  const secretHash = await argon2.hash(tokenSecret);

  // 3. Return the Combined Token (for user) and Hash (for DB)
  return {
    token: `${tokenId}.${tokenSecret}`, // User gets "id.secret"
    hash: secretHash,
    id: tokenId
  };
};
```

#### 2. Saving to DB (`auth.service.js`)

```javascript
export const createSession = async (identityId, userAgent, ip) => {
  const { token, hash, id } = await createRefreshToken();

  // Save using the ID as the Primary Key or a dedicated column
  await prisma.session.create({
    data: {
      id: id, // Store the ID explicitly if using it as PK
      token_hash: hash, // Store the hash of the secret
      identity_id: identityId,
      user_agent: userAgent,
      expires_at: new Date(Date.now() + 7 * 24 * 60 * 60 * 1000) // 7 days
    }
  });

  return token; // Send "id.secret" to user
};
```

#### 3. Verifying the Token (`auth.controller.js -> refresh`)

```javascript
export const refresh = catchAsync(async (req, res) => {
  const { refreshToken } = req.cookies;
  if (!refreshToken) throw new AppError("No token provided", 401);

  // 1. Split the Token
  const [tokenId, tokenSecret] = refreshToken.split('.');
  if (!tokenId || !tokenSecret) throw new AppError("Invalid token format", 401);

  // 2. Lookup by ID (Fast & Direct)
  const session = await prisma.session.findUnique({
    where: { id: tokenId }, // Using the ID part!
    include: { identity: true }
  });

  if (!session) throw new AppError("Session not found", 401);

  // 3. Verify the Secret (Secure Check)
  const isValid = await argon2.verify(session.token_hash, tokenSecret);
  if (!isValid) throw new AppError("Invalid token signature", 401);

  // 4. Check Expiry
  if (new Date() > session.expires_at) {
    await prisma.session.delete({ where: { id: tokenId } });
    throw new AppError("Session expired", 401);
  }

  // ... Issue new Access Token ...
});
```

### Summary
1.  **Format:** Use `TokenID.TokenSecret`.
2.  **DB:** Store `TokenID` (Index) and `Argon2(TokenSecret)`.
3.  **Lookup:** Find by `TokenID`. Verify `TokenSecret`.

This is the standard, secure way to handle hashed token lookups without compromising performance or security.

can the current store id be set in the header - x-store-id?

**Yes, absolutely.**

Using a header like `x-store-id` (or `x-tenant-id`) is a very common pattern in multi-tenant SaaS applications, especially if you want to allow users to switch stores quickly without re-authenticating (getting a new token).

**However, there is a major security catch.**

### The Security Risk
You **cannot** trust the header blindly. If I am a hacker logged in as "User A", I can send `x-store-id: store-B`. If your backend blindly trusts the header, I just accessed someone else's store.

### How to implement it safely

You need a piece of middleware that acts as a **Gatekeeper**. It must verify: *"Is this User actually a member of the Store ID sent in the header?"*

#### 1. The Middleware (`tenant.middleware.js`)

This middleware sits **after** your Auth (`protect`) middleware.

```javascript
import { AppError } from '../utils/AppError.js';
import { catchAsync } from '../utils/catchAsync.js';
import { PrismaClient } from '@prisma/client';

const prisma = new PrismaClient();

export const validateStoreHeader = catchAsync(async (req, res, next) => {
  // 1. Check if header exists
  const storeId = req.headers['x-store-id'];

  if (!storeId) {
    // For Platform Admins, maybe this is optional? 
    // For Store Users, this is mandatory.
    if (req.user.systemRole === 'PLATFORM_ADMIN') return next();
    
    return next(new AppError('x-store-id header is missing', 400));
  }

  // 2. Platform Admins bypass the check (God Mode)
  if (req.user.systemRole === 'PLATFORM_ADMIN') {
    req.tenantId = storeId; // Inject into request context
    req.userRole = 'OWNER'; // Treat admin as owner
    return next();
  }

  // 3. Regular Users: VERIFY MEMBERSHIP IN DB
  // This is the "Cost" of using headers - a DB query on every request.
  const membership = await prisma.storeMember.findUnique({
    where: {
      storeId_identityId: {
        storeId: storeId,
        identityId: req.user.id // From the 'protect' middleware
      }
    }
  });

  if (!membership) {
    return next(new AppError('You do not have access to this store', 403));
  }

  // 4. Inject Context
  req.tenantId = storeId;
  req.userRole = membership.role; // OWNER, CASHIER, etc.

  next();
});
```

---

### 2. Architecture Comparison: Header vs. Token

Since we previously discussed putting `tenantId` inside the JWT, you now have two architectural choices. You must pick one.

#### Option A: The "Stateless" Token Approach (Previously Discussed)
*   **How:** User calls `/select-store` -> Gets JWT containing `tenantId`.
*   **Pros:** **Fast**. No database check needed on every request. The token *is* the proof of membership.
*   **Cons:** Hard to switch stores. User must re-negotiate a token to switch contexts.

#### Option B: The "Header" Approach (Your Question)
*   **How:** User Login -> Gets "Global User Token". Frontend sends `x-store-id` on every request.
*   **Pros:** **Flexible**. User can open Tab A (Store 1) and Tab B (Store 2) using the same auth token.
*   **Cons:** **Slower**. You MUST hit the database (`storeMember.findUnique`) on *every single API call* to verify they belong to that store.

---

### 3. Recommendation for your App

**Use a Hybrid Approach.**

1.  **For the POS (Cashier App):** Use **Option A (Token)**.
    *   Cashiers don't switch stores often. They log in to a specific register.
    *   Speed is critical. You don't want a DB lookup slowing down a checkout.

2.  **For the Admin Dashboard (Manager/Owner):** Use **Option B (Header)**.
    *   Owners might manage 5 stores and want to switch dropdowns quickly.
    *   The 20ms penalty for the DB lookup is acceptable for an admin interface.

### How to configure the Hybrid Route

**`src/modules/sales/sales.routes.js`**

```javascript
import { protect } from '../../shared/middleware/auth.middleware.js';
import { validateStoreHeader } from '../../shared/middleware/tenant.middleware.js';

// Route for Admin Dashboard (Uses Header)
router.get('/dashboard-stats', 
  protect, 
  validateStoreHeader, // <--- Checks DB, allows context switching
  salesController.getStats
);

// Route for POS (Uses Token)
router.post('/checkout', 
  protect, // <--- Token already has tenantId, no DB check needed
  salesController.createOrder
);
```

### Performance Optimization (Advanced)
If you go with the **Header** approach and are worried about database load:
*   **Cache the Membership:** When `validateStoreHeader` runs, check Redis (`GET membership:user_123:store_abc`).
*   If not found, check DB, then set Redis.
*   This makes the Header approach almost as fast as the Token approach.

i was asking specifically for the refresh endpoint 

**No, you should NOT set `x-store-id` in the header for the refresh endpoint.**

Here is why: **The Refresh Endpoint's job is to ISSUE an Access Token.**

If you send `x-store-id` in the header, you are implying you already *have* an authenticated context for that store, which is circular logic because you are asking for a token *to get* that context.

### The Problem: Ambiguity
1.  **User A** has a refresh token valid for **Store 1** AND **Store 2**.
2.  If they call `POST /refresh` with `x-store-id: store-1`, the server issues a token for Store 1.
3.  **BUT:** What if the user *switched tabs* and is now looking at Store 2? The refresh logic might run in the background (auto-renew) and suddenly issue a token for Store 1, breaking their Store 2 session.

### The Correct Pattern: Passing Context in Body (Explicit Intent)

The **Refresh Request** is a transactional operation. You are trading a Refresh Token for an Access Token. The *context* of that trade should be explicit in the **Body**.

**Request:**
```http
POST /api/auth/refresh
Cookie: refreshToken=abc...xyz
Content-Type: application/json

{
  "currentStoreId": "store-123" // Optional context
}
```

**Controller Logic:**
```javascript
export const refresh = catchAsync(async (req, res) => {
  const { refreshToken } = req.cookies;
  
  // 1. Verify Refresh Token (Identity Level)
  const session = await authService.verifySession(refreshToken);
  
  // 2. Check Context (From Body)
  const { currentStoreId } = req.body;

  if (currentStoreId) {
     // Verify they still belong to this store
     // This prevents a user from refreshing a token for a store they were kicked out of
     const member = await tenantService.validateMembership(session.identity.id, currentStoreId);
     
     // Issue Store-Scoped Token
     const token = await signToken({
        sub: session.identity.id,
        sid: session.id,
        tenantId: currentStoreId,
        role: member.role
     });
     
     return res.json({ token });
  }

  // Fallback: If no storeId provided (e.g., initial load), give them a "Selection Token"
  const tempToken = await signToken({ 
     sub: session.identity.id, 
     scope: 'STORE_SELECTION' 
  });
  
  res.json({ token: tempToken, requireSelection: true });
});
```

### Why Body > Header for Refresh?
1.  **Explicit:** Headers like `x-store-id` are often set globally by Axios interceptors. You don't want your global interceptor accidentally forcing a store context onto the refresh logic if the user is currently on the "Select Store" screen.
2.  **Safety:** By putting it in the body, you force the frontend developer to *consciously decide* which store they are trying to refresh access for.

### Exception: Platform Admin
If the user is a **Platform Admin**, the `currentStoreId` is ignored (or used purely for logging), because they get a Super-User token regardless.

should the protect middleware also check for expiration or is it handled automatically when decoding the jwt

**It is handled automatically.**

The `jsonwebtoken` (and `jose`) library automatically checks the `exp` (Expiration Time) claim when you call `jwt.verify()` or `jwtVerify()`.

If the token is expired:
1.  **Library:** Throws a `TokenExpiredError`.
2.  **Middleware:** Your `catchAsync` wrapper catches this error.
3.  **Global Error Handler:**
    *   **Development:** Returns full stack trace.
    *   **Production:** Returns `{ status: 'fail', message: 'jwt expired' }` (or similar).

### What you need to handle manually
While expiration is automatic, there are **two** edge cases you must handle manually inside `protect`:

1.  **Revocation (Logout/Password Change):**
    *   The token is *valid* (signature & expiry are fine), but the user logged out or changed their password 5 minutes ago.
    *   **Solution:** Check `iat` (Issued At) against `user.passwordChangedAt`. If `iat < passwordChangedAt`, throw 401.

2.  **Deleted User:**
    *   The token is valid, but the user was deleted from the DB 1 minute ago.
    *   **Solution:** Check `await prisma.identity.findUnique({ where: { id: decoded.id } })`. If `null`, throw 401.

### Example Implementation

```javascript
export const protect = catchAsync(async (req, res, next) => {
  // 1. Get token
  // ...

  // 2. Verify token (Automatic Expiry Check happens here!)
  const decoded = await promisify(jwt.verify)(token, process.env.JWT_SECRET);

  // 3. Check if user exists (Manual)
  const currentUser = await prisma.identity.findUnique({ where: { id: decoded.id } });
  if (!currentUser) {
    return next(new AppError('The user belonging to this token does no longer exist.', 401));
  }

  // 4. Check if user changed password after the token was issued (Manual)
  if (currentUser.changedPasswordAfter(decoded.iat)) {
    return next(new AppError('User recently changed password! Please log in again.', 401));
  }

  // GRANT ACCESS
  req.user = currentUser;
  next();
});
```
